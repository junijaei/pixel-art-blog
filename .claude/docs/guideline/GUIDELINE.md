# Notion CMS 기반 블로그 프로젝트 가이드

## 1. 문서 목적

이 문서는 **Notion을 CMS로 사용하는 개인 블로그 프로젝트**에 대한 전반적인 설계 의도와 기술적 가이드라인을 설명한다.
AI는 이 문서를 기준으로 프로젝트의 맥락을 이해하고, 이후 발생하는 질문이나 구현 논의에서 **일관된 판단 기준**을 유지해야 한다.

- 프로젝트의 **방향성, 제약 조건, 선택 배경**을 공유하는 데 목적이 있다.

---

## 2. 프로젝트 개요

### 2.1 목표

- 유지비 0원
- SEO 친화적인 개인 블로그
- 글 작성·수정이 최대한 간편한 운영 방식
- 디자인 및 렌더링 커스터마이징이 가능한 구조

### 2.2 기본 페이지 구성

- 글 목록 페이지
- 글 상세 페이지
- About 페이지

### 2.3 추가 기능 범위

- 검색
- 카테고리 분류
- 태그 분류
- 반응형 디자인
- 다크 모드 지원
- 댓글 기능은 필수가 아님
  (도입 시 giscus를 고려하되, 기본 요구사항에는 포함하지 않음)

---

## 3. 기술 스택

### 3.1 프레임워크 및 런타임

- **Next.js**
  - SEO 대응
  - ISR(Incremental Static Regeneration) 활용
  - 정적 페이지 기반 블로그 구조에 적합

### 3.2 스타일링 및 UI

- **Tailwind CSS**
  - 반응형 디자인
  - 다크 모드 스타일 제어

- **shadcn/ui**
  - 기본 UI 컴포넌트 구성

- **next-themes**
  - 다크 모드 / 라이트 모드 전환 관리

### 3.3 검색·분류 기능

- 검색, 카테고리, 태그는 **클라이언트 단에서 단순 구현**
- 서버 검색 엔진(Algolia 등)은 사용하지 않음
- 빌드 시 메타데이터를 기반으로 필터링 가능한 구조를 전제로 함

---

## 4. CMS 설계: Notion 사용

### 4.1 CMS 선택 기준

CMS(Content Management System)는 다음 요구를 충족해야 한다.

- 글 작성과 수정이 빠르고 직관적일 것
- IDE 실행 없이 콘텐츠 관리 가능할 것
- 잦은 수정에도 심리적 부담이 적을 것
- 버전 관리가 과도하게 노출되지 않을 것

### 4.2 Notion을 CMS로 선택한 이유

- 이미 익숙한 작성 환경
- 마크다운 기반 작성 가능
- 웹 UI로 즉시 수정 가능
- 커밋 단위 수정 관리에서 오는 부담 제거

MDX 파일 기반 블로그는 다음 이유로 제외한다.

- 매 글 수정 시 IDE 실행 필요
- 사소한 수정도 Git 히스토리로 남는 구조
- 글 관리보다 개발 도구 사용 비중이 커짐

## 5. Notion 블록 렌더링 전략

### 5.1 기본 원칙

- Notion 콘텐츠는 **Markdown 변환을 거치지 않는다**
- Notion API가 제공하는 **block schema를 그대로 사용**
- 각 block type을 **React 컴포넌트로 직접 매핑**한다

### 5.2 렌더링 접근 방식

- Notion API
  - Page 조회
  - Block 목록 조회

- Block 단위 처리
  - `paragraph`
  - `heading_1`, `heading_2`, `heading_3`
  - `image`
  - `code`
  - `callout`
  - 기타 필요한 block은 점진적으로 확장

각 block은 다음 구조를 따른다.

- `block.type` 기준 분기
- type별 전용 React 컴포넌트
- `has_children`을 고려한 재귀 렌더링

### 5.3 이 방식의 의도

- Notion의 구조적 의미 보존
- 디자인 및 레이아웃 커스터마이징 자유도 확보
- 마크다운 변환 과정에서 발생하는 정보 손실 방지
- 장기적인 유지보수 안정성 확보

---

## 6. 데이터 및 캐싱 전략

### 6.1 이미지 처리

- Notion 이미지 블록은 빌드 시 로컬 파일로 저장
- 판단 기준
  - `last_edited_time` 비교

- 이미 저장된 이미지 중 변경되지 않은 파일은 재저장하지 않음

### 6.2 페이지 캐싱 전략

- **ISR(Incremental Static Regeneration)** 사용
- 콘텐츠 변경 시에만 재빌드
- 이미지 변경 여부는 별도의 로직으로 판단

---

## 7. 배포 전략

- **Vercel**
  - 무료 플랜 기준
  - 자동 CDN 제공
  - Next.js와의 높은 궁합

- 별도의 서버 비용, CDN 비용을 발생시키지 않는 것을 전제로 함

---

## 8. 프로젝트 전반의 설계 철학

- 콘텐츠 관리의 중심은 “개발 편의성”이 아니라 “작성 경험”
- 라이브러리 의존보다 **구조에 대한 제어권**을 우선
- 초기 구현 비용보다 **장기 유지보수와 확장성**을 중시
- 블로그는 애플리케이션이 아니라 **지속적으로 수정되는 콘텐츠 시스템**으로 취급

---

## 9. AI 응답 시 기대 기준

AI는 이 프로젝트와 관련된 질문에 대해 다음을 전제로 판단한다.

- Notion은 CMS이며, 단순 데이터 소스가 아님
- 마크다운 변환 기반 접근은 기본 해법이 아님
- Block schema 기반 렌더링이 중심 설계임
- “빠른 구현”보다 “설계 일관성”을 우선한다
